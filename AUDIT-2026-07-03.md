## Critical

### 1. Phase null + out-of-range bug — `generatePrompt` emits `"phase null of null."` or `"phase 5 of 3."`
**Severity:** Critical · **Complexity:** 1 · **Triage: ship now**

When `phasedExecution` is on but `totalPhases`/`currentPhase` are empty, `numOrNull` returns `null` and the sprint line appends `", you'll be executing only phase null of null."` — garbage in the generated prompt. Even with a finite check, `currentPhase > totalPhases` (e.g. `5 of 3`) and `0 of 0` still emit invalid clauses. The UI's `updatePhaseHint` already warns on `current > total` (`index.html` lines 993–997); `generatePrompt` should enforce the same.

**Location:** `index.html` lines 867–869.

**Solution:**
```js
if (state.phasedExecution === true &&
    Number.isInteger(state.currentPhase) && state.currentPhase > 0 &&
    Number.isInteger(state.totalPhases) && state.totalPhases > 0 &&
    state.currentPhase <= state.totalPhases) {
  sprintLine += ", you'll be executing only phase " + state.currentPhase + " of " + state.totalPhases;
}
```
`Number.isInteger` (not just `Number.isFinite`) — phases are whole numbers.

**Risks:** None. Strictly narrows the condition; existing valid inputs (positive integers, `current ≤ total`) behave identically.

**Required tests:**
- `phasedExecution=true, totalPhases=null, currentPhase=null` → sprint line ends with `.` (no phase clause).
- `phasedExecution=true, totalPhases=3, currentPhase=null` → no phase clause.
- `phasedExecution=true, totalPhases=3, currentPhase=2` → `", you'll be executing only phase 2 of 3."`.
- `phasedExecution=true, totalPhases=3, currentPhase=5` → no phase clause (out of range).
- `phasedExecution=true, totalPhases=0, currentPhase=0` → no phase clause.
- `phasedExecution=true, totalPhases=2.5, currentPhase=1` → no phase clause (non-integer).
- `phasedExecution=false` → no phase clause regardless of phase values.
- Phase hint UI still warns when fields empty or out of range (regression guard).

## High

### 2. Clipboard rejection unhandled + "Copied!" feedback lies
**Severity:** High · **Complexity:** 2 · **Triage: ship now**

`copyToClipboard` primary path: `navigator.clipboard.writeText(text).then(...)` — no `.catch`. On rejection (secure-context permission denial), the promise rejects unhandled, no fallback runs, no error feedback. Separately, `handleCopy` always shows "Copied!" regardless of the boolean `copyToClipboard` resolves with, so the fallback-2 path (returns `false`) still claims success.

Note: `file://` and non-localhost HTTP make `navigator.clipboard` **undefined** (not rejected) — those users hit fallback 1 (`execCommand`), which works. The broken path is the narrower secure-context permission-denial case. No data loss either way (text stays in `#promptOutput`). Downgraded from Critical to High after cross-review confirmed the real-world `file://` flow works.

**Location:** `index.html` lines 1015–1044 (`copyToClipboard`), 1047–1061 (`handleCopy`).

**Solution:** extract fallback into a named helper, add `.catch`, use the returned boolean for feedback, and replace the intrusive `alert()` with the existing `setWarning` banner so the UI stays consistent:
```js
function fallbackExecCommand(text) {
  try {
    const ta = document.createElement("textarea");
    ta.value = text;
    ta.setAttribute("readonly", "");
    ta.style.position = "fixed";
    ta.style.top = "-9999px";
    ta.style.opacity = "0";
    document.body.appendChild(ta);
    ta.select();
    const ok = document.execCommand("copy");
    document.body.removeChild(ta);
    if (ok) return true;
  } catch (e) { /* fall through */ }
  const pre = document.getElementById("promptOutput");
  const range = document.createRange();
  range.selectNodeContents(pre);
  const sel = window.getSelection();
  sel.removeAllRanges();
  sel.addRange(range);
  setWarning("Clipboard unavailable. Select the prompt and press Cmd/Ctrl+C.", "warn");
  return false;
}

function copyToClipboard(text) {
  if (navigator.clipboard && navigator.clipboard.writeText) {
    return navigator.clipboard.writeText(text)
      .then(function () { return true; })
      .catch(function () { return fallbackExecCommand(text); });
  }
  return Promise.resolve(fallbackExecCommand(text));
}

function handleCopy() {
  const state = getState();
  const text = generatePrompt(state);
  if (state.task === null) flashTaskLabel();
  const btn = document.getElementById("copyBtn");
  const original = btn.textContent;
  copyToClipboard(text).then(function (ok) {
    btn.textContent = ok ? "Copied!" : "Copy failed";
    btn.classList.toggle("copied", ok);
    setTimeout(function () {
      btn.textContent = original;
      btn.classList.remove("copied");
    }, 1500);
  });
}
```

**Risks:** `execCommand` is deprecated; fallback may stop working in future browsers. Acceptable — it's a fallback, primary path stays `navigator.clipboard`. `setWarning` is shared with import/export; a copy-fail warning could be overwritten by a subsequent import warning. Acceptable — copy feedback also lives on the button text.

**Required tests:**
- Mock `navigator.clipboard.writeText` to resolve → button shows "Copied!", `copied` class added, reverts after 1.5s, no warning banner.
- Mock `navigator.clipboard.writeText` to reject → fallback `execCommand` runs; if it returns `true`, "Copied!" shows; no unhandled rejection in console.
- Mock both `navigator.clipboard` absent and `execCommand` returning `false` → button shows "Copy failed", no `copied` class, warning banner shows (not `alert`), `#promptOutput` selected so Cmd/Ctrl+C still works.
- `state.task === null` → task legend still flashes (regression guard).
- Real browser smoke: copy works over HTTP and `file://` (manual).

### 3. Template import accepts wrong value types — silent data corruption
**Severity:** High · **Complexity:** 2 · **Triage: defer**

`importTemplate` validates the envelope (`app`, `templateVersion`, `state` object) but not state field types. Runtime-confirmed: `state.projectName = {evil:true}` → field renders `"[object Object]"`; `state.caveman = "yes"` → silently unchecked (string !== `true`); `state.totalPhases = "abc"` → stripped to `""`. No warning shown. Forward-compatibility is intended for unknown keys, not for wrong-typed known keys.

**Why deferred:** requires a hand-edited or malformed template — edge case, not a security vector (`JSON.parse` is inert, no code execution). The forward-compat contract already says unknown keys are ignored. Revisit if users report it. Listed here so a contributor can pick it up.

**Location:** `index.html` lines 1122–1173 (`applyState`).

**Solution:** type-guard the setters:
```js
const setVal = function (el, value) {
  if (!el || value === undefined || value === null) return;
  if (typeof value !== "string" && typeof value !== "number") return;
  el.value = String(value);
};
const setChecked = function (el, value) {
  if (!el || value === undefined) return;
  el.checked = value === true;
};
```
For numeric phase fields, coerce via `numOrNull` and reject non-finite:
```js
setVal(form.elements.totalPhases,
  Number.isFinite(state.totalPhases) ? state.totalPhases : "");
setVal(form.elements.currentPhase,
  Number.isFinite(state.currentPhase) ? state.currentPhase : "");
```

**Risks:** Templates exported by older versions with slightly different types would now be rejected silently for those fields. Mitigation: only guard the known fields listed in AGENTS §4; unknown keys already ignored. Document the tightening in `changelog.md`.

**Required tests:**
- Import `state.projectName = {evil:true}` → field stays empty, no `"[object Object]"`.
- Import `state.caveman = "yes"` → checkbox stays in its pre-import state (not silently unchecked if it was checked).
- Import `state.totalPhases = "abc"` → field empty, phase hint warns.
- Import `state.totalPhases = 3` → field `3` (regression).
- Import `state.audit = "not-an-object"` → audit checkboxes unchanged, no throw.
- Import valid v1.0.0 template → restores exactly (regression).
- `dbVersion` mismatch warning still fires (regression).

### 5. `database.js` load failure = blank page
**Severity:** High · **Complexity:** 1 · **Triage: ship now**

`<script src="database.js">` is in `<head>`. If the file is missing or misnamed, `DATABASE` is undefined → `init()` throws → blank page with only a console error. The logo has an `onerror` handler; the data file does not.

**Location:** `index.html` line 8.

**Solution:**
```html
<script src="database.js"
        onerror="document.body.innerHTML='<p style=&quot;padding:2rem;font-family:sans-serif;color:#991B1B&quot;>Failed to load database.js. Place it next to index.html and refresh.</p>'"></script>
```
And guard `init`:
```js
function init() {
  if (typeof DATABASE === "undefined") return;
  // ...existing body
}
```

**Risks:** The inline `onerror` handler uses an inline style and would be blocked by a strict CSP. If CSP is added later (see issue 12), move the handler into `init`'s guard and render the message via `textContent`. Minimal risk for now.

**Required tests:**
- Serve `index.html` without `database.js` → user-facing error message visible, no uncaught throw in console.
- Serve with `database.js` present → normal load (regression).
- Rename `database.js` → same error path.
- Verify the guard returns early before any DOM mutation.

function render() {

### 8. Double render on text inputs (`input` + `change` listeners)
**Severity:** Medium · **Complexity:** 1 · **Triage: defer**

`form.addEventListener("input", render)` and `form.addEventListener("change", render)` both fire. For text inputs, `input` fires per keystroke and `change` fires on blur → `render` runs twice per field edit. The explicit `phasedExecution` change listener (line 1225) also calls `render()`, and checkboxes fire both `input` and `change`, so some controls still double-render even after dropping the global `change` listener.

**Why deferred:** invisible to the user. Micro-performance on a 23-input form generating ~1KB of text. Pure internals — adds nothing visible. Listed so a contributor can pick it up.

**Location:** `index.html` lines 1225–1231.

**Solution:** the explicit `phasedExecution` listener should only call `refreshVisibility()`; let the global `input` listener handle `render`. Then drop the global `change` listener — `input` fires for text, select, and checkbox in all modern browsers.
```js
form.elements.phasedExecution.addEventListener("change", function () {
  refreshVisibility();
  // render() is handled by the global input listener below,
  // which also fires for checkbox toggles.
});
form.addEventListener("input", render);
```
This matches AGENTS §6 (visibility centralized in `refreshVisibility`).

**Risks:** Some old browsers fired `input` inconsistently for `<select>`; modern browsers (the project's stated runtime) are fine. If a regression appears, restore a `change` listener only for `select` elements.

**Required tests:**
- Type in project name → `#stateDump` updates once per keystroke (not twice). Verify by counting `JSON.stringify` calls via a spy or by checking the dump's content.
- Change a `<select>` → `render` fires once.
- Toggle a checkbox → `render` fires once.
- Toggle `phasedExecution` → `refreshVisibility` fires (audit/phase blocks show/hide) AND `render` fires once (via the global `input` listener), not twice.

### 14. `applyState` doesn't validate `projectType` / `stack`
**Severity:** Medium · **Complexity:** 2 · **Triage: ship now**

`applyState` calls `setVal(form.elements.projectType, state.projectType)` with no membership check. `rolePhrase`'s fallbacks (`database.js` lines 55–59) silently absorb unknown values: unknown stack → falls back to `"Full stack"` template; unknown projectType → `tech = ""` (tech prefix dropped). So a template exported with an older `dbVersion` (or a hand-edited one) can set `projectType: "Vue"` and the role line silently drops the tech prefix with no warning.

**Location:** `index.html` lines 1134–1137.

**Solution:** validate against `DATABASE.projectTypes` / `DATABASE.stacks`; if absent, keep the current value (don't silently fall through):
```js
if (DATABASE.projectTypes.indexOf(state.projectType) !== -1) {
  setVal(form.elements.projectType, state.projectType);
}
if (DATABASE.stacks.indexOf(state.stack) !== -1) {
  setVal(form.elements.stack, state.stack);
}
```

**Risks:** templates with renamed-but-equivalent types now leave the dropdown at its current value instead of applying the stale name. Acceptable — the user re-selects. The `dbVersion` mismatch warning (already implemented) still fires to alert the user.

**Required tests:**
- Import `state.projectType = "Vue"` (not in DB) → dropdown unchanged, no silent tech-prefix drop in the generated role line.
- Import `state.stack = "Mobile"` (not in DB) → dropdown unchanged.
- Import `state.projectType = "Swift"` (valid) → dropdown sets to Swift (regression).
- Import `state.stack = "Frontend"` (valid) → dropdown sets to Frontend (regression).
- `dbVersion` mismatch warning still fires (regression).

### 10. `exportTemplate` lacks try/catch
**Severity:** Low · **Complexity:** 1 · **Triage: defer**

`exportTemplate` calls `window.prompt`, `Blob`, `URL.createObjectURL`, `a.click`, `URL.revokeObjectURL`. None wrapped. If `Blob` or `URL.createObjectURL` throws (rare, but possible in hardened browsers), the error is unhandled and the user sees nothing.

**Why deferred:** `Blob`/`URL.createObjectURL` basically never throw in modern browsers. Theoretical. Listed so a contributor can pick it up.

**Location:** `index.html` lines 1096–1120.

**Solution:**
```js
function exportTemplate() {
  const name = window.prompt("Template name:", "Untitled");
  if (name === null) return;
  try {
    const envelope = { /* ...existing... */ };
    const blob = new Blob([JSON.stringify(envelope, null, 2)], { type: "application/json" });
    const url = URL.createObjectURL(blob);
    const a = document.createElement("a");
    const safeName = /* ...existing... */;
    a.href = url;
    a.download = safeName + "-" + timestampStamp() + ".json";
    document.body.appendChild(a);
    a.click();
    document.body.removeChild(a);
    URL.revokeObjectURL(url);
  } catch (e) {
    setWarning("Export failed: " + (e && e.message ? e.message : "unknown error") + ".", "error");
  }
}
```

**Risks:** None. Pure additive guard.

**Required tests:**
- Mock `URL.createObjectURL` to throw → warning banner shows error, no uncaught exception.
- Normal export → file downloads, no warning (regression).
- Cancel the prompt → no export attempted (regression).

### 12. Optional CSP for hosted deployments
**Severity:** Low · **Complexity:** 2 · **Triage: defer (CSP only)**

If the tool is ever hosted, add a CSP header. The README trust warning for `database.js` (the original other half of this issue) is folded into issue 6 and ships now.

**Why deferred:** adding CSP prematurely breaks the page — inline `<style>` and the inline `onerror` on `<img>` would be blocked. Only ship CSP after extracting inline handlers/style to external files. Not needed for `file://` (the stated runtime per AGENTS §1.2).

**Location:** optional `<meta>` in `index.html`.

**Solution:** only after extracting inline `<style>` and the `<img onerror>` handler:
```html
<meta http-equiv="Content-Security-Policy"
      content="default-src 'self'; img-src 'self' data:; style-src 'self'">
```

**Risks:** Adding CSP prematurely breaks the page — inline `<style>` and `onerror` would be blocked. Only ship CSP after the inline handlers are gone.

**Required tests:**
- If CSP added: load `index.html` over HTTP, confirm no CSP violations in console, logo loads, copy works, export downloads. Run the full manual checklist from AGENTS §7.

`applyState` calls `setVal(form.elements.projectType, state.projectType)` with no membership check. `rolePhrase`'s fallbacks (`database.js` lines 55–59) silently absorb unknown values: unknown stack → falls back to `"Full stack"` template; unknown projectType → `tech = ""` (tech prefix dropped). So a template exported with an older `dbVersion` (or a hand-edited one) can set `projectType: "Vue"` and the role line silently drops the tech prefix with no warning.

**Location:** `index.html` lines 1134–1137.

**Solution:** validate against `DATABASE.projectTypes` / `DATABASE.stacks`; if absent, keep the current value (don't silently fall through):
```js
if (DATABASE.projectTypes.indexOf(state.projectType) !== -1) {
  setVal(form.elements.projectType, state.projectType);
}
if (DATABASE.stacks.indexOf(state.stack) !== -1) {
  setVal(form.elements.stack, state.stack);
}
```

**Risks:** templates with renamed-but-equivalent types now leave the dropdown at its current value instead of applying the stale name. Acceptable — the user re-selects. The `dbVersion` mismatch warning (already implemented) still fires to alert the user.

**Required tests:**
- Import `state.projectType = "Vue"` (not in DB) → dropdown unchanged, no silent tech-prefix drop in the generated role line.
- Import `state.stack = "Mobile"` (not in DB) → dropdown unchanged.
- Import `state.projectType = "Swift"` (valid) → dropdown sets to Swift (regression).
- Import `state.stack = "Frontend"` (valid) → dropdown sets to Frontend (regression).
- `dbVersion` mismatch warning still fires (regression).
